## Rules now implemented on-chain

### Brute creation

- First 3 brutes per wallet: free via `BrutusRegistryV2.createBaseBrute(bytes32 metadataHash)`.
- 4th+ brutes: paid via `createExtraBrute(bytes32 metadataHash)`.
- Current extra brute price: `0.01 BNB` for first extra, doubles per extra.
- Extra brute BNB split:
  - 50% to the real Flap vault: `0x4e7a2406a623dB7DdB0a93748732Bb7D95390A02`
  - 50% to burn/dead address: `0x000000000000000000000000000000000000dEaD`

### Combat winner rewards

- Operator records a fight winner with `BrutusCombatRewardsV2.recordCombatWin(fightId, winner, winnerBruteId)`.
- Winner can claim exactly `0.001 BNB` with `claimCombatReward(fightId)`.
- Loser cannot claim.
- Winner cannot claim unless wallet holds at least `10,000` of token `0x473159256928B25f6B0E7cd75d49EbD4ac977777`.
- Each fight can only be claimed once.
- The real vault now forwards its tax BNB to CombatRewardsV2 because vault `rewardReceiver` was updated to `0xc6737672eC7351aFC6607d565510436086d4e8E9`.

### Paid 1v1 / tournaments

- `BrutusArenaEscrowV2` supports paid 1v1 challenges.
- Creator deposits stake.
- Opponent accepts with matching stake.
- Resolver signs winner.
- Winner claims the pot.
- Current feeBps deployed as `0`, so winner takes all while testing.

## Local database reality

The current backend uses local SQLite:
`prisma/dev.db`

This is not Railway/Postgres yet. I added wallet/on-chain columns and a migration:
`prisma/migrations/20260627133000_add_wallet_onchain_brutes/migration.sql`

The local API now blocks infinite local brute creation:
- requires `walletAddress` on create
- max 3 local brutes per wallet
- a 4th local brute requires on-chain proof fields (`onChainBruteId` + `createTxHash`)

For production/Railway, use Postgres by changing `DATABASE_URL` and `provider = "postgresql"` before production migration.
